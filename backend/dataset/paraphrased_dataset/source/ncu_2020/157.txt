UDP 是一種不提供可靠性的資料傳輸協定，UDP 不進行複雜的控制，只提供非連結 導向式的通訊服務。即使在網路壅塞的情況下，UDP 仍不會控制傳送量來避免網路壅塞。 就算封包遺失，也不會進行重送，也沒有調整封包送達順序的功能。如果需要進行這些 控制，通常由TCP/IP 的應用層來處理。 每一個UDP 訊息都稱為使用者資料包 （User Datagram），而且是由兩個部分所組成： 一是識別傳送和接收應用程式的短表頭，另一是帶有要送出的承載資料，圖2-1 為User Datagram 格式說明。 圖2-1 User Datagram 格式 資料來源：（Comer, 2015） UDP 表頭的前面兩個欄位皆是16 位元的通訊協定埠號碼。UDP 來源埠記載對應傳 送者應用程式的Port number，而UDP 目的埠用來記載對應接收此訊息應用程式的Port number，UDP 訊息長度欄位，用來設定UDP 訊息的總長度。以下圖2-2 為DNS Query 的 UDP 表頭範例。 UDP 表頭包含：  Source Port: 62062  Destination Port: 53  Length: 42  Checksum: 0x63fc 圖2-2 DNS Query 的UDP 表頭範例 資料來源：本研究 總結來說，UDP 協定本身的處理很簡單，直接透過TCP/IP 的IP 層傳送封包，不須 建立傳輸路徑，沒有錯誤處理 （Error Control） 與流量控制 （Flow Control），所以運作 速度很快，適合以下用途。  總封包較少的通訊，如DNS、SNMP 等。  影片及聲音等需要即時性的多媒體通訊。  在特定網路中，以特定的應用程式通訊。  需要一傳多的通訊，如廣播、群播。
