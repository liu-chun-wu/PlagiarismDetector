在提及Oracle 之資料庫時，通常會涉及其高可用及災難復原系統。首 先，在高可用性的部份，以RAC (Real Application Cluster)資料庫架構最為 常見，一般是兩個或兩個以上的節點共用一組磁碟陣列系統，正常時可透過 多個節點同時執行查詢的SQL 語句，當單節點出現故障時，會將服務 Failover 至另一台節點主機，但磁碟陣列資料有問題時則完全失去功能，且 如無備份系統的話，資料將完全無法回復(Hussain et al., 2013)。為說明該系 統之架構，本研究將以下圖5 為例，加以進行說明： 圖5  RAC 標準架構 資料來源：Hussain (2013) 在該圖中包含的為兩台標準的12C RAC 架構，用戶或是Application 的 連線是透過DNS 名稱解析進入SCAN(Single Client Access Name )，經由 VIP(Virtual IP )轉換實際連結到該Node 的Listener 服務監聽器，然後再由 Instance 實例(Oracle 在該Node 的記憶體結構)，主要有兩大要素 ：共享記 憶體SGA，及一組背景處理程序Background Process 執行SQL 語句，把執 行輸出給用戶或是Application。 災難備援系統Data Guard：它是一套提供可以創建一個或多個備用資 料庫的工具，可自動控管切換，保護服務避免、故障、災難、人為錯誤和資 損壞的影響(Hart and Jesse, 2004)。營運持續(Business Continuity)和災難復原 (Disaster recovery)，是絕大部分企業IT 管理者的最優先考量，全天24 小時 的營運環境，已經變成大多數企業經營的必要條件，而Oracle Data Guard 正是其中一個非常有效率的解決方案(姚文琳、姚大鋼，2006)。 Data Guard 的運作方式是透過主庫與備庫雙方的監聽器，透過網路傳 輸，並保持資料一致性，備用資料庫可建置在任何可透過網路連線的數據機 房，當然兩地網路傳輸效率的好壞，會影響資料的延遲性。另外因為的故障 而無法使用主資料庫時，Data Guard 可切換任何一台備用資料庫，來成為 主資料庫的角色。透過此功能，也可以盡量縮小停機時間與減少數據丟失的 可能性。 Data Guard 是使用交易日誌(Redo Log)來維持主備資料庫間資料的一 致性，其運作原理可參考圖6，當主資料庫有交易發生時，Redo Log 會被 產生並寫至本機 Redo Log Files，這些資料也會傳輸到備用資料庫上，並且 被還原套用Apply 保持資料與主資料庫一致，所以Data Guard 的優點也正 好可以彌補RAC 架構無法做到的資料保護，其機制如下圖所示： 圖6  12C Active Data Guard 資料來源：Oracle (2015) Array-Based Replication 第三方磁碟陣列複製技術，是指用磁碟陣列內 建功能，即時的將資料複製至同品牌類型的磁碟陣列主機上，以確保資料的 目的能有兩份歸檔，但此方式與 Data Guard 相比，所耗損的系統資源較大， 僅能以資料區塊做差異比對後，減少單次複製傳輸量，且無法確保切換後的 資料完整性，所以如果預算充裕，還是建議使用Oracle 原生的Data Guard 方式做異地備援系統。
