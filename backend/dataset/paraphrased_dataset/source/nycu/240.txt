確立自動化測試首先要判斷的標的是軟體測試中測試案例的可 重復利用性，軟體工程除了針對某些特殊的用戶需求，重復利用過程 中的組件在結構和功能上都不太可能發生很大的變化，因此一般採用 測試團隊的測試方法生成的測試案例都是可重復利用的。然而自動化 測試並不是萬能的，其無法涵蓋所有的手動測試，但是卻可以減輕人 工手動測試的負擔，自動化測試包含了幾項特點[14]： 1. Ability to run two or more specified test cases：必須可同時執行多個 案例的測試，而不只是一次只能測一個，要能獨立完成複數的整合性 測試而不只是讓測試人員反覆操作執行每次的測試項目。 2. Ability to run a subset of all the automated test cases：系統必須可以針 對各式功能或相關聯的設定進行測試，讓開發人員可以修改部份的程 式碼，便可以針對此相關功能進行驗證。 3. No intervention is needed after launching the tests：當執行自動化測試 開始時，便不需要於測試過程中有人為動作介入，不然便失去自動化 的初衷，自動化測試必須依據已經準備好的項目所需參數與環境進而 透過script 執行。 4. Automatically sets-up and/or records the relevant test environment parameters：自動化系統於測試前一般是維持在預設值，之後透過 script 來針對所需要的測試條件自動去設定相關環境參數，以此來針 對不同案例進行不同環境參數設定。 5. Runs the test cases：當測試案例套用自動化後，便可以依照其需求 執行自動化測試。 6. Captures the relevant results：自動化測試案例中最難的部分便是 - 12 - debug，由於測試人員必須先確定是否是自動化程式的問題還是測試 產品本身的功能問題，因此於每一個自動化執行的過程中必須包含紀 錄每一個執行步驟及設定的log 檔，透過log 能了解是基於哪一個項 目的步驟出現問題，並且讓開發者透過log 檔來驗證產品問題。如果 自動化系統可以大量收集資訊便能大幅縮短issue 的驗證與確認時間 和解決問題的時間。 7. Compares actual with expected results and flags differences：每一個測 試案例都會有其對應的預期結果，在程式自動化之後都必須可以明確 的判斷案例測試成功或失敗，因此必須要清楚標示與定義該功能的預 期結果與實際測試結果來相互比對，並計算出統計結果。 8. Analyzes and reports pass/fail for each test case and for the test run：測 試報告內每一個項目必須要有記載詳細能對應的log 檔，測試步驟的 記錄便可以針對各個測試個案來做分析。
