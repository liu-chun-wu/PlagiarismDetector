本研究在實際測試成果方面主要利用前述章節所提之專屬指令來傳送資訊給予 BMC 系統，讓BMC 系統得以藉由指令獲取相應的資訊。本研究主要依賴無線網路來傳 送資訊而非伺服器上的實體線路，故需要一個穩定正常的網路環境並給予BMC 系統一 組網路IP 位址；且因本研究主要假設環境為智慧居家，故以區域網路為主要測試環境。 在此，本研究於樹梅派第三代系統以及另一台Ubuntu 系統主機中，放置測試腳本 並設定其相應資訊以及執行次數後執行，並觀測BMC 系統是否有收到相應的資訊並將 其數據導入管理的資訊中。本研究測試腳本，請參考測試腳本之範本如圖4-4，主要必 須提供以下資訊： 1. 循環次數 設定腳本指令在迴圈中所需要執行的次數，設定數值為0 時，腳本會持續執行。 2. BMC 系統網路IP 位址 設定BMC 系統的網路IP 位置，外部系統和裝置可經由BMC 系統的網路IP 位置 傳送資訊給BMC 系統。 3. 網路功能代碼（Net Function Code） 除了IPMI 智慧平台管理介面已定義的Function Code 之外，可由系統開發端和客戶 間協商出新的Function Code，用以區分指令的種類和簡易管理，例如以代碼40h 來 表示主要用於某廠牌專用的指令集。 4. 指令代碼（Command Number） 和Net Function Code 搭配使用，為指令本身所屬之代碼。 5. 感知器或裝置身分代碼（Sensor Number） 在BMC 系統中所設定的感知器或裝置ID 代碼。 6. 網路延遲容許值（Network Delay Time） 本研究針對網路傳輸可能造成延遲以及不穩定的情況所增加的一個參數，讓BMC 系統在網路延遲的情況下，不至於立即將暫延遲傳送資訊的管理項目視為其數值遺 失或錯誤，而會先以參考此參數為延遲容許值；若超過延遲容許值依舊未取得新數 據才會視為數值遺失或錯誤；而數值遺失或錯誤時，在延遲容許值之內將以上一次 有效數值為依據。本研究建議可依據網路環境以及設備裝置的重要性和急迫性來調 整其數值，在一般正常網路環境下建議可以三秒為基本參數，並加以參考設備之重 要性和急迫性來予以調整。 7. 傳送資料長度（Data Length）： 預計傳送的資料長度，亦即本次指令之下所要傳輸資料的長度並以Byte 為資料計 算的基礎單位。 8. 裝置或感知器資料（Device Data／Sensor Value） 裝置或感知器所傳送的資料，根據IPMI 智慧平台管理介面目前的標準，一個指令 大小不超過256 Bytes 即可，本腳本範例僅以傳送一個Byte 作為代表。 圖4-4 腳本範例 資料來源：本研究整理
