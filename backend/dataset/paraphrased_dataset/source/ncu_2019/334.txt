本節主要說明本研究在分析系統架構過程中以及實作開發系統時曾遭遇的數點問 題，於此章節闡述其原由以及本研究採用之解決方案： 1. 網路品質差異導致數據異常或取得時間不一 在現今一般既有的伺服器管理機制上，受管控的感知器（Sensor）或裝置（Device） 其大多依賴實體線路的傳送；在依賴實體線路的機制下，資訊傳遞的延遲和錯誤率 都是相對低的，即便有某些裝置或管理項目需要增加額外的處理或調整，或者需要 再加上一定程度的重送及驗證機制，但實務上大多會滿足資訊快速傳遞和低錯誤率 的要求。而本研究在資訊的傳送上，為了跳脫實體線路的束縛，在研究設計上需要 依賴無線網路或行動通訊來傳遞資訊，而依賴無線網路或行動通訊在傳送速率上和 資訊準確性上就不如依賴實體線路了，故需要一定的除錯和延遲容許機制。同時現 今大多的伺服器管理機制多為要求一秒內掃描過所有的感知器或裝置至少一次並 獲得其資訊，亦即一秒為一個監控循環或掃描週期；但隨著感知器和裝置的多元發 展和考量實際應用面上的操作，需要更有彈性的去調整其資訊保留的時效性。本研 究的設計中，在專屬／客製化指令中設計了網路延遲時間的參數，使用者可以利用 設定網路延遲時間這個參數來調整資訊傳遞時延遲的容許值，在延遲容許值內，其 數值會以上一次有效的數值為依據，而不會因為新的資訊或數值無法在一秒內亦即 一個掃描週期內取得，進而被BMC 系統判定為錯誤或監控項目無法取得相應資訊。 在網路延遲時間這個參數的設定上，本研究建議可依據網路環境以及設備裝置的重 要性和急迫性來調整其數值，在一般正常網路環境下建議以三秒為基本參數，並加 以參考設備之重要性和急迫性來予以調整。 2. 同一時間連結主機數量有限 一般伺服器在管理上會限定建立session 連線的上限值，用以避免同一時間過多的連 線可能會增加系統的負擔。在本研究的架構下並未挑戰其session 極限，因目前設定 值為15 個sessions 連線為最大值，已足夠多數環境下使用，在此列出以供參考。 3. 未使用之Sensor Number／Sensor ID 散置 在目前一般伺服器的BMC 系統架構中大多並未有開放給使用者自行運用的部分， 所以在規劃Sensor Number／Sensor ID 時大多是依照客戶需求或系統開發者自行定 義，通常不會將Sensor Number／Sensor ID 用盡，但同時並不會刻意留下連續的區 塊提供給使用者或廠商自行利用。針對此問題只需要在系統開發時將Sensor Number ／Sensor ID 上做些規劃和預留，便可留下連續的Sensor Number／Sensor ID 區域供 使用者自行使用，同時也方便系統管理和未來運用，例如Sensor Number／Sensor ID 編號80h 至A0h（16 進制）開放給使用者或廠商自行利用。 4. 使用者自行加入或建立及修改管理項目 BMC系統會將受管理的感知器以及裝置資訊存放於SDR（Sensor Data Record，感知 器資訊紀錄）中，並以此份檔案為管理依據，使用標準的IPMI智慧平台管理介面指 令雖然可用於修改和調整現有資訊，但便利性和使用上較為不便，因此，當要建立 或修改SDR時，大多還是依靠系統廠或業者自行開發的軟體來操作，先將SDR完整 取出新增修改後再加以載入。本研究主要探討利用IPMI智慧平台管理介面運用於智 慧居家之可行性，故並無開發用以建立或修改SDR所用之新軟體，改以在系統開發 階段事先預留可供自行運用的Sensor Number／Sensor ID區塊為主，並設定管理項目 所需之基本設定，在此僅列出以供參考。
